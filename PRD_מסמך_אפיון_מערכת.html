1. תקציר מנהלים

1.1 מהו הפרויקט?

פרויקט זה מבצע אוטומציה לתהליך הטיפול בשגיאות היזון חוזר פנסיוני המבוצע כיום באופן ידני על ידי מנהלות תיקים. המערכת תבצע:

  • קריאה - דוחות שגיאות מהמסלקה הפנסיונית
  • זיהוי - אילו שגיאות דורשות טיפול
  • הכנה - טיוטות מייל אוטומטיות
  • שחרור זמן - מנהלות תיקים יטפלו רק במקרים מורכבים

1.2 מדוע אנחנו צריכים את זה?

❌ מצב נוכחי

  • מנהלות תיקים משקיעות שעות רבות בשבוע בסקירה ידנית של קבצי שגיאות
  • שגיאות חוזרות ועוקבות דפוסים צפויים
  • סיכון לטעות אנוש בזיהוי הגורם האחראי
  • תגובה מאוחרת לנושאים דחופים

✅ מצב עתידי (עם אוטומציה)

  • צמצום 90% בעבודה ידנית
  • טיוטות מייל מוכנות באותו היום
  • אפס שגיאות בזיהוי גורמים אחראיים
  • מנהלות תיקים ממוקדות במשימות בעלות ערך גבוה בלבד

1.3 סיכום השקעה

פרמטר פירוט
לוח זמנים 8-12 שבועות (פיילוט → ייצור)
החזר השקעה (ROI) חיסכון של 15-20 שעות שבועיות לכל מנהלת תיק
טכנולוגיה אוטומציה n8n + עיבוד Python + אינטגרציה Gmail
רמת סיכון נמוכה (השקה מדורגת, נדרש אישור אנושי)

💡 ערך מיידי

שבוע ראשון: מנהלת תיק תקבל 20 טיוטות מייל מוכנות במקום לכתוב אותן ידנית - חיסכון של 4 שעות.

חודש ראשון: חיסכון של כ-16 שעות עבודה = יום עבודה שלם בחודש לכל מנהלת תיק.

2. הבעיה העסקית וההזדמנות

2.1 התהליך הנוכחי (ידני)

📅 כל שבוע:

  1. מנהלת תיק מקבלת קובץ Excel מהמסלקה הפנסיונית (מאות/אלפי שורות)
  2. סוקרת ידנית כל שגיאה
  3. מזהה מי האחראי (מעסיק מול קופת גמל)
  4. כותבת מיילים נפרדים לכל גורם
  5. עוקבת אחר המשך טיפול באופן ידני
מדד ערך נוכחי
השקעת זמן 15-20 שעות/שבוע
שיעור טעויות 5-10% (גורם אחראי שגוי, שגיאות שהוחמצו)
זמן תגובה 2-5 יום

2.2 השפעה עסקית של התהליך הנוכחי

עלויות:

  • עלות עבודה גבוהה לעבודה חוזרת
  • פתרון שגיאות מושהה = תשלומים מושהים
  • השפעה על שביעות רצון לקוחות
  • סיכון רגולטורי

הזדמנויות שהוחמצו:

  • מנהלות תיקים יכולות להתמקד ביחסי לקוחות
  • זיהוי פרואקטיבי של בעיות
  • ניתוח מגמות ומניעה

2.3 ההזדמנות

85-90% מהשגיאות עוקבות דפוסים סטנדרטיים
16 שעות חיסכון חודשי למנהלת תיק
100% דיוק בזיהוי אחריות

4. יכולות ותכונות המערכת

4.1 תכונות ליבה

תכונה 1: קליטת נתונים חכמה

מה היא עושה:

  • קוראת קבצי Excel/XML מהמסלקה
  • מסננת להציג רק שגיאות רלוונטיות (4 סוגי סטטוס מוגדרים)
  • פיילוט ראשוני: שבוע אחרון בלבד
  • ייצור: כל הנתונים מאז תאריך העלייה לייצור

ערך עסקי:

  • אפס הכנת קבצים ידנית
  • איכות נתונים עקבית
  • אין רשומות שמוחמצות

תכונה 2: סיווג שגיאות חכם

מה היא עושה:

  • מתאימה קודי שגיאה לחוקים מוגדרים מראש
  • מחשבת כמה זמן כל שגיאה פתוחה (מונה שבועי)
  • קובעת גורם אחראי:
    • מעסיק (הנפוץ ביותר)
    • קופת גמל (במקרים מיוחדים)
    • דורש סקירה ידנית (מקרים מורכבים)

ערך עסקי:

  • דיוק 100% בהקצאה
  • הסלמה אוטומטית לשגיאות ותיקות
  • אחריות ברורה

תכונה 3: מעקב משך זמן והסלמה

לוגיקת הסלמה אוטומטית:

שבוע פעולה
שבוע 0 שגיאה חדשה → אין פעולה (המתנה לשבוע הבא)
שבוע 1 שליחת מייל ראשון לגורם האחראי
שבוע 2 שליחת תזכורת עם הפרדה (חדשות מול חוזרות)
שבוע 3 מייל פנימי למנהלת תיק (נדרש מעקב טלפוני)
שבוע 4 מייל פנימי למנהלת תיק + מנהל
שבוע 5+ מייל פנימי למנהלת תיק + מנהל + מנכ"ל

ערך עסקי:

  • אין שגיאות ש"נופלות בין הכיסאות"
  • הסלמה אוטומטית
  • נראות להנהלה על נושאים כרוניים

תכונה 4: יצירת טיוטות מייל אוטומטית

מה היא עושה:

  • יוצרת טיוטות מייל ב-Gmail (לא נשלחות אוטומטית)
  • מיילים חיצוניים (למעסיקים):
    • מייל אחד לכל מעסיק (מאחד את כל השגיאות שלו)
    • תבנית מקצועית עם פירוט שגיאות
    • קובץ Excel מצורף עם רשימת שגיאות מלאה
  • מיילים פנימיים (לצוות):
    • סיכום שגיאות הדורשות תשומת לב
    • פריטי פעולה לפי עדיפות

ערך עסקי:

  • חיסכון 90% בזמן כתיבת מיילים
  • תקשורת מקצועית עקבית
  • מנהלת תיק סוקרת ומאשרת לפני שליחה

תכונה 5: אנליטיקה ודיווח

מה היא עושה:

  • מייצרת דוח סיכום שבועי:
    • סך שגיאות לפי סוג
    • פילוח לפי מעסיק/קופת גמל
    • ניתוח גיל (כמה זמן שגיאות פתוחות)
    • ניתוח מגמות (משתפר/מחמיר)
  • שולח דוח לדשבורד הפורטל

ערך עסקי:

  • נראות להנהלה
  • זיהוי מעסיקים/קופות בעייתיים
  • החלטות מבוססות נתונים

4.2 לוגיקה מיוחדת: חוקי Override

📋 עבור קודי שגיאה 4, 5, 15, 93:

המערכת בודקת נתונים היסטוריים (חודשים קודמים)

שאלה: האם זה עובד בהצלחה בעבר?

  • ✓ כן → אחריות = קופת גמל (לא מעסיק)
  • ✗ לא → אחריות = מעסיק

למה זה חשוב:

  • מונע האשמה שגויה של מעסיקים
  • פתרון מהיר יותר (פנייה לגורם הנכון מיד)

5. מסע המשתמש וזרימת העבודה

5.1 זרימה שבועית של מנהלת תיק

❌ לפני (תהליך ידני)

יום שני:
  • הורדת קובץ (10 דקות)
  • פתיחת Excel, סקירת 500 שורות (2 שעות)
  • זיהוי שגיאות (שעה)
  • חיפוש אנשי קשר מעסיקים (30 דקות)
  • כתיבת 20 מיילים נפרדים (4 שעות)
  • מעקב בגיליון אלקטרוני (30 דקות)

סה"כ: ~8-9 שעות

✅ אחרי (תהליך אוטומטי)

יום שלישי בוקר:
  • קבלת התראה: "20 טיוטות מייל מוכנות לסקירה"
  • פתיחת Gmail, סקירת טיוטות (30 דקות)
  • עריכת 2-3 מקרים מורכבים (15 דקות)
  • לחיצה על "שלח" בכל הטיוטות המאושרות (5 דקות)

סה"כ: ~50 דקות (חיסכון 90%!)

6. ארכיטקטורה טכנית

6.1 רכיבי מערכת (הסבר לא-טכני)

חשבו על המערכת כקו ייצור במפעל:

תחנה תפקיד טכנולוגיה
1. תחנת קלט מקבלת את הקובץ השבועי אוטומטית
בדיקת איכות: מוודאת פורמט נתונים		 חיבור REST API לפורטל "השקט שלך"
2. תחנת עיבוד קוראת כל שורת שגיאה
מיישמת חוקים עסקיים
מחשבת משך זמן
מקבצת שגיאות דומות		 סקריפטי Python רצים ב-n8n
3. תחנת החלטה מתאימה קוד שגיאה לפעולה
קובעת גורם אחראי
בוחרת תבנית מייל		 טבלת מסד נתונים + לוגיקת Python
4. תחנת פלט יוצרת טיוטות מייל
מצרפת קבצי Excel
רושמת כל פעולה		 Gmail API (קריאה/כתיבה לטיוטות בלבד)
5. תחנת דיווח מייצרת דוח סיכום
שולחת לדשבורד		 API push לפורטל

6.2 מערך טכנולוגי (למנהלים טכניים)

רכיב טכנולוגיה מטרה
תזמון n8n (Self-Hosted) אוטומציה של זרימות עבודה, תזמון
עיבוד Python 3.11+ ניתוח נתונים, לוגיקה עסקית
אחסון נתונים PostgreSQL חוקים, טבלאות מיפוי, לוגים
מייל Gmail API יצירת טיוטות (OAuth2, scope מוגבל)
אינטגרציה REST APIs חיבור לפורטל "השקט שלך"
אבטחה Static IP + OAuth2 גישה מאובטחת, מסלול ביקורת
ניטור לוגים מובנים של n8n מעקב שגיאות, ביצועים

6.3 אבטחה ופרטיות

🔒 הגנת נתונים:

  • ✓ כל המידע המזהה אישית (ת.ז., שמות) נשאר בתוך מערכות מאובטחות
  • ✓ כתובת IP סטטית לכל חיבורי API
  • ✓ אימות OAuth2 (אין סיסמאות שמורות)
  • ✓ טיוטות מייל בלבד (Gmail scope מוגבל - לא יכול לשלוח אוטומטית)
  • ✓ לוג ביקורת מלא של כל הפעולות

📋 תאימות רגולטורית:

  • תקנות פרטיות: כל הנתונים מטופלים בהתאם לחוק הגנת הפרטיות הישראלי
  • אין נתונים נשלחים לשירותי ענן חיצוניים (למעט Gmail/Google Workspace מאושרים)

8. מדדי הצלחה

8.1 מדדים תפעוליים

מדד יעד איך נמדוד
חיסכון בזמן צמצום 85-90% שעות שהושקעו ביצירת טיוטות מייל
שיעור שגיאות <2% סיווג שגוי ביקורת ידנית על מדגם (50 מקרים/שבוע)
זמן תגובה טיוטות מוכנות באותו יום זמן מקובץ זמין → טיוטות מוכנות
שיעור אישור >95% טיוטות נשלחות כמות שהן % טיוטות שנשלחו ללא עריכה

8.2 מדדים עסקיים

מדד יעד השפעה
זמן פתרון שגיאות צמצום 30% פתרון מהיר יותר לנושאי לקוחות
שגיאות כרוניות (5+ שבועות) צמצום 50% תהליך הסלמה טוב יותר
שביעות רצון מנהלות תיקים דירוג 8/10+ יותר זמן לעבודה בעלת ערך גבוה
חיסכון בעלויות 15-20 שעות/שבוע/מנהלת תיק חישוב החזר השקעה

8.3 מדדים טכניים

99%+ זמן פעילות מערכת (רץ בימי שלישי בבוקר)
<10 דקות זמן עיבוד ל-1,000 רשומות
<1% בקשות API כושלות

9. סיכונים והפחתתם

9.1 סיכונים בעדיפות גבוהה

סיכון השפעה הסתברות הפחתה
הקצאת גורם אחראי שגוי גבוהה נמוכה - בדיקות נרחבות בפיילוט
- לוגיקת override לקודים מיוחדים
- אישור אנושי לפני שליחה
מגבלות קצב Gmail API בינונית נמוכה - עיבוד אצווה
- לוגיקת ניסיון חוזר מובנית
- התראות ניטור
פריצת פרטיות נתונים גבוהה נמוכה מאוד - IP סטטי בלבד
- scopes מוגבלים ב-OAuth
- אין שיתוף נתונים חיצוני
- לוגי ביקורת
השבתת מערכת ביום ג' בבוקר בינונית נמוכה - תהליך גיבוי ידני
- ניטור התראות
- SLA להתאוששות באותו יום

9.2 סיכונים בעדיפות בינונית

סיכון הפחתה
תבניות זקוקות לעדכונים תכופים - ספריית תבניות במסד נתונים (עדכונים קלים)
- בקרת גרסאות
שינויי API מ"השקט שלך" - תיעוד API
- תהליך הודעה על שינויים
- בדיקות תאימות לאחור
התנגדות לאוטומציה - הדרכת מנהלות תיקים
- עלייה מדורגת
- הדגמת חיסכון בזמן

10. נספח: מילון מונחים

למנהלים לא-טכניים

מונח הסבר בעברית פשוטה
API דרך לשני מערכות מחשב לדבר זו עם זו אוטומטית (כמו קו טלפון בין מערכות)
טיוטת מייל (Draft) מייל שהוכן על ידי המערכת אבל לא נשלח - ממתין ב-Gmail לאדם לסקור ולאשר

IP סטטי כתובת אינטרנט קבועה לחיבורים מאובטחים (כמו מספר טלפון קבוע)
